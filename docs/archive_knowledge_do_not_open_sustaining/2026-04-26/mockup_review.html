Same account, two Pipelines, decided by FeatureEngine per-FeatureKey. Neither Deployment named a Pipeline.
Worked example B — one Deployment, multiple timeframes (3m + 1w):
A momentum strategy needing 3-minute bars for entry timing AND weekly bars for trend context doesn't pick a Pipeline. It hands {3m.close[0], 1w.high[0], …} to FeatureEngine, which:
  • Resolves both keys to alpaca-premium-1 (1m base supports 3m derivation; 1d base supports 1w derivation).
  • Subscribes once at the Pipeline (1m bars), derives 3m by aggregating 3 × 1m and 1w by aggregating ~5 × 1d.
  • Dedups against any other Deployment requesting the same FeatureKeys.
Capability-matching rule: per-FeatureKey, a Pipeline's finest native timeframe must be ≤ that key's required timeframe. Roll coarser, never finer.
Worked example C — one Deployment, multiple Pipelines:
A strategy needs 5m equity bars + OPRA option IV + corporate-events news. No single Pipeline serves all three. Under the old "one Pipeline per Deployment" rule this would be impossible. Under FeatureEngine mediation, it's the normal case:
  • 5m.close[0] → resolved to alpaca-premium-1
  • 5m.option_iv[0] → resolved to polygon-options (would need to be added)
  • stream.news.corporate_events → resolved to news-vendor-1
FeatureEngine subscribes to three Pipelines on this Deployment's behalf. The Deployment never knows.
